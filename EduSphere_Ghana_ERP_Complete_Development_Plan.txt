SECTION 1: EXECUTIVE SUMMARY
================================================================================

EduSphere Ghana ERP is a cloud-ready, multi-tenant, multi-campus, GES-compliant
School ERP and SaaS platform serving Nursery, KG, Primary, JHS, SHS, TVET,
Colleges, and Universities across Ghana and beyond.

Two major enhancements are included in this master plan:

A) DYNAMIC SCORING ENGINE
   - Each school independently configures class score components (exercises,
     homework, projects, practicals, tests, observations, and custom fields).
   - Each school sets the class score maximum and exam score maximum.
   - The system auto-scales both to any target weight (e.g., 50/50, 30/70, 40/60).
   - Teachers enter raw scores per component; the system aggregates and scales.
   - Fully configurable per school, per level, per subject, per term.

B) DYNAMIC PROMOTION ENGINE
   - Implements Ghana's basic school practice: end-of-year promotion is decided
     in Term 3, using the cumulative/average performance across Term 1, Term 2,
     and Term 3 (GES does not impose one single rigid national percentage for
     basic-school promotion — schools set internal promotion policy guided by
     continuous assessment; EduSphere lets each school configure its own rule,
     e.g. annual average of all 3 terms, or "average for 2 of 3 terms including
     Term 3", patterns commonly used by Ghanaian basic schools).
   - Separate, independently configurable rule sets per level (Nursery/KG,
     Primary, JHS, SHS), since promotion logic differs by level in Ghana.
   - JHS 3 (Basic 9) and SHS terminal-year candidates are automatically
     excluded from internal promotion and routed to BECE/WASSCE + CSSPS
     placement instead of school-decided promotion.
   - Repetition tracking with a configurable repeat limit (GES practice
     generally allows a pupil to repeat a class at most once before further
     review).
   - Headteacher/Principal review-and-approve workflow before results are
     published to parents, with full audit trail.

C) DRAG-AND-DROP SCHOOL WEBSITE BUILDER
   - Every school gets its own subdomain/custom domain website.
   - Built-in page builder with drag-and-drop sections/blocks.
   - Components: hero, about, news, gallery, events, admissions, contact, etc.
   - SEO settings, branding, color themes, and typography controls.
   - Publish/unpublish, revision history, preview mode.
   - Content feeds directly from ERP data (news, events, staff, gallery).

Tech Stack:
  Backend:   PHP 8.3+, Laravel 12, MySQL 8, Redis, REST API
  Frontend:  Bootstrap 5, HTML5, CSS3, JavaScript ES6, Alpine.js, jQuery
  Tools:     Chart.js, DomPDF, QR Generator, Laravel Sanctum, PWA
  Builder:   GrapesJS (open-source drag-and-drop page builder, JS)
  DevOps:    XAMPP (dev), Apache/Nginx (prod), SSL, Cloudflare-ready

--------------------------
- PHP/Laravel ecosystem is mature, well-documented, and widely supported.

SECTION 4: SYSTEM REQUIREMENTS SPECIFICATION (SRS)
================================================================================

4.1 FUNCTIONAL REQUIREMENTS
-----------------------------

FR-001: Multi-Tenant Architecture
  The system SHALL support multiple schools on one platform with strict data
  isolation via school_id on all tenant tables and tenant middleware.

FR-002: Dynamic Scoring Engine
  FR-002a: Each school SHALL configure class score components independently
           (name, maximum marks, weight/order, active/inactive).
  FR-002b: Schools SHALL set the class score total (e.g., 50 or 100) and exam
           score total (e.g., 100).
  FR-002c: Schools SHALL set the target scaled weight for class score and exam
           score (e.g., class=50%, exam=50%; or class=30%, exam=70%).
  FR-002d: The system SHALL auto-calculate:
           - Component totals per student per subject per term.
           - Scaled class score = (raw class total / class max) * class weight.
           - Scaled exam score  = (raw exam mark / exam max) * exam weight.
           - Grand total = scaled class score + scaled exam score.
  FR-002e: Configuration SHALL be per school, with optional overrides per
           level (Primary, JHS, SHS) and per subject.
  FR-002f: Teachers SHALL enter raw scores; the system scales automatically.
  FR-002g: Score entry SHALL support spreadsheet-style grid, Excel/CSV upload,
           copy-paste, autosave, and missing-score alerts.
  FR-002h: A configuration preview SHALL show teachers and admins exactly how
           scores will be calculated before publishing.

FR-002P: Dynamic Promotion Engine
  FR-002P-a: Each school SHALL configure a Promotion Configuration per academic
             level (Nursery/KG, Primary, JHS, SHS, Tertiary), since GES does
             not enforce one fixed national promotion percentage at the basic
             level — promotion policy is set internally by each school based
             on continuous assessment performance.
  FR-002P-b: For Nursery–JHS2 (Basic 1–8), the engine SHALL compute, at the
             close of Term 3, an annual performance figure from the three
             terms' published grand totals, using the school's selected
             METHOD:
               Method 1 — Annual Average: average of Term 1 + Term 2 + Term 3
                          grand totals (weights per term configurable, default
                          equal weight 1:1:1).
               Method 2 — Two-of-Three: average of the pupil's best 2 of the
                          3 terms, provided Term 3 is one of the two counted
                          (mirrors common Ghanaian basic-school practice).
               Method 3 — Subject Pass Count: pupil must obtain at least the
                          configured pass mark in at least N of the subjects
                          studied, AND meet the minimum overall average.
  FR-002P-c: The school SHALL set a Promotion Threshold (e.g., overall average
             ≥ 40%, ≥ 50%, or ≥ 60%) and, optionally, a per-subject minimum
             pass mark and a minimum number of subjects that must be passed.
  FR-002P-d: JHS 3 (Basic 9) pupils and SHS terminal-year (graduating) students
             SHALL be automatically excluded from internal promotion
             processing; instead they are flagged "BECE Candidate" or
             "WASSCE Candidate" and routed to the External Examination &
             CSSPS/Tertiary Placement module.
  FR-002P-e: For SHS (Forms 1–3 / Years 1–3), promotion to the next form SHALL
             be configurable using continuous assessment + house-exam
             aggregate, with optional core/elective subject pass-count rules
             and conduct/attendance gating, reflecting practice at Ghanaian
             senior high schools.
  FR-002P-f: The system SHALL track repeats per student per class and enforce
             a configurable repeat limit (default: 1 repeat per class level)
             before flagging the case for Headteacher/Principal manual review.
  FR-002P-g: The engine SHALL auto-generate a Promotion List at the end of
             Term 3, classifying every pupil as Promoted, Conditional/On
             Probation, or Repeat, with the figures and rule that produced
             each classification fully visible (no "black box" decisions).
  FR-002P-h: The Class Teacher and Headteacher/Principal SHALL review and
             approve the Promotion List (with ability to override individual
             decisions with a mandatory reason) before it is published.
  FR-002P-i: On publish, the system SHALL automatically create each promoted
             student's enrollment record in the next class for the new
             academic year (student_enrollments.promoted_from_class_id),
             and SHALL automatically re-enroll repeating students in the
             same class.
  FR-002P-j: Promotion status SHALL appear on the Term 3 report card
             ("Promoted to Basic 5", "Repeated", "BECE Candidate", etc.).

FR-003: School Website Builder
  FR-003a: Each school SHALL have a website accessible at their subdomain or
           custom domain.
  FR-003b: The website builder SHALL use drag-and-drop with pre-built blocks:
           Hero, About, Admissions, News Feed, Events, Gallery, Staff, Contact,
           Custom HTML, Custom Image, Video Embed, Testimonials, Call-to-Action.
  FR-003c: Schools SHALL be able to add/remove/reorder pages.
  FR-003d: SEO settings (title, meta description, og:image) per page.
  FR-003e: Branding: logo, colors, fonts, favicon.
  FR-003f: Content blocks can pull live data from ERP (news posts, events,
           gallery images) or be manually authored.
  FR-003g: Publish/unpublish with preview mode and revision history.
  FR-003h: Website is publicly accessible WITHOUT login; ERP is protected.

FR-004 through FR-034: [All 34 modules from the master prompt — see Section 12
  Phase Plan for implementation details per phase]

SECTION 5: USER STORIES & USE CASES
================================================================================

5.1 DYNAMIC SCORING ENGINE — USER STORIES
------------------------------------------

US-DS-001 (School Admin):
  "As a School Admin, I want to define the class score components for each level
  so that teachers know exactly what to enter and the system calculates correctly."
  Acceptance:
    - Admin can add component (name, max marks, display order).
    - Admin can set class score total and exam score total.
    - Admin can set scale weights (e.g., class=50, exam=50 out of 100).
    - System validates that component max marks sum <= class score total.
    - Preview shows sample calculation before saving.

US-DS-002 (Teacher):
  "As a Teacher, I want to enter raw scores for each component per student and
  see the system automatically calculate the total class score and final grade."
  Acceptance:
    - Score entry grid shows columns per configured component.
    - Real-time running total shown per student row.
    - System flags out-of-range entries instantly.
    - Autosave drafts every 30 seconds.

US-DS-003 (HOD):
  "As an HOD, I want to verify teacher-entered scores before they are sent to
  the headteacher for approval."
  Acceptance:
    - HOD sees a subject-level broadsheet with all students and components.
    - HOD can flag errors and return to teacher with comments.
    - HOD approval is timestamped and logged.

US-DS-004 (Parent):
  "As a Parent, I want to see my child's report card showing both the class score
  breakdown and the exam score, and understand how the final grade was achieved."
  Acceptance:
    - Report card shows each component and marks.
    - Shows scaled class score, scaled exam score, and total.
    - Grade and position shown with the grading scale key.

US-DS-005 (Super Admin):
  "As a Super Admin, I want to see that each school's scoring configuration is
  independent and does not affect other schools."
  Acceptance:
    - Score configurations are scoped by school_id.
    - No cross-school data visible in any configuration screen.

5.2 SCHOOL WEBSITE BUILDER — USER STORIES
------------------------------------------

US-WB-001 (School Admin):
  "As a School Admin, I want to build our school website by dragging and dropping
  sections so that I can create a professional website without coding knowledge."
  Acceptance:
    - GrapesJS editor loads within the ERP admin panel.

US-WB-002 (School Admin):

  Acceptance:
    - Dynamic News Block pulls from the ERP news/announcements module.
    - Dynamic Events Block pulls from the ERP calendar/events module.
    - Gallery block pulls from ERP document/gallery module.

US-WB-003 (School Admin):
  "As a School Admin, I want to set our school colors, logo, and fonts on the
  website to match our branding."
  Acceptance:
    - Branding panel: upload logo, set primary/secondary/accent colors.

US-WB-004 (Super Admin):
  "As a Super Admin, I want each school's website to be isolated so that School A
  cannot edit or view School B's website."
  Acceptance:
    - Website content is scoped by school_id.
    - Custom domain routing maps domain to school safely.

7.5 DYNAMIC SCORING ENGINE TABLES (NEW — DETAILED)
----------------------------------------------------

scoring_configurations
  id, school_id, campus_id (nullable),
  level (ALL / Primary / JHS / SHS / TVET / Tertiary),
  subject_id (nullable: NULL = applies to all subjects at this level),
  academic_year_id (nullable),
  name (e.g. "Standard SHS Configuration"),
  class_score_max        -- e.g., 50 or 100 (raw total teacher enters up to)
  class_score_weight     -- e.g., 50 (what it scales to on report card)
  exam_score_max         -- e.g., 100 (raw exam mark)
  exam_score_weight      -- e.g., 50 (what exam scales to on report card)
  grand_total            -- always class_score_weight + exam_score_weight
  rounding_method        -- ROUND / FLOOR / CEIL
  decimal_places         -- 0, 1, or 2
  is_active, is_default,
  created_by, updated_by, created_at, updated_at, deleted_at

  CONSTRAINTS:
    CHECK (class_score_weight + exam_score_weight = grand_total)
    CHECK (grand_total IN (100, 50) OR grand_total > 0)

  EXAMPLE CONFIGURATIONS:
    GES Standard JHS:
      class_score_max=50, class_score_weight=50,
      exam_score_max=100, exam_score_weight=50, grand_total=100
      Meaning: teacher enters up to 50 for class score (no scaling needed
      since 50/50=1.0 * 50 = 50), and up to 100 for exam (100/100 * 50 = 50).

    Alternative 30/70:
      class_score_max=30, class_score_weight=30,
      exam_score_max=100, exam_score_weight=70, grand_total=100

    Alternative where class score is out of 100 but scales to 50:
      class_score_max=100, class_score_weight=50,
      exam_score_max=100, exam_score_weight=50, grand_total=100
      Formula: scaled_class = (raw_class/100)*50; scaled_exam=(raw_exam/100)*50

score_components
  id, school_id, scoring_configuration_id (FK),
  name (e.g. Exercise 1, Homework, Project Work, Test, Practical, Observation),
  max_marks     -- e.g., 10, 5, 20
  display_order -- for ordering on score sheet and report card
  is_active,
  is_required (bool: must be entered before submission),
  created_by, updated_by, created_at, updated_at, deleted_at

  CONSTRAINT: SUM(max_marks) per scoring_configuration_id <= class_score_max
  (enforced at application level with validation)

  EXAMPLE for class_score_max=50:
    Exercise 1:  max 10
    Exercise 2:  max 10
    Homework:    max 5
    Project:     max 15
    Class Test:  max 10
    TOTAL:       50 ✓

student_scores
  id, school_id, student_id (FK), class_id (FK), stream_id (FK nullable),
  subject_id (FK), term_id (FK), academic_year_id (FK),
  scoring_configuration_id (FK),
  teacher_id (FK users),

  -- Component scores stored as JSON for flexibility:
  component_scores (JSON)
    e.g. {"component_1_id": 8, "component_2_id": 9, "component_3_id": 14, ...}

  -- Computed fields (stored for performance, recalculated on demand):
  raw_class_total     -- sum of component scores

8.1 CORE CALCULATION LOGIC
----------------------------

The scoring engine is implemented as a Laravel Service class:
  App\Services\Scoring\ScoringEngineService

CALCULATION FORMULA:
  1. raw_class_total    = SUM of all entered component scores for student
  2. scaled_class_score = ROUND((raw_class_total / class_score_max) * class_score_weight, decimal_places)
  3. scaled_exam_score  = ROUND((raw_exam_score / exam_score_max) * exam_score_weight, decimal_places)
  4. grand_total        = scaled_class_score + scaled_exam_score
  5. grade              = lookup in grading_scale_items where grand_total BETWEEN min_score AND max_score

EXAMPLE CALCULATIONS:

  SCHOOL A (class=50/50, exam=100/50, total=100):
    Components: Ex1=8/10, Ex2=9/10, HW=4/5, Project=12/15, Test=7/10
    raw_class_total    = 8+9+4+12+7 = 40
    scaled_class_score = (40/50)*50 = 40.0
    raw_exam_score     = 72
    scaled_exam_score  = (72/100)*50 = 36.0
    grand_total        = 40 + 36 = 76 → Grade: B2

  SCHOOL B (class=100/30, exam=100/70, total=100):
    Components sum to 70 out of 100
    scaled_class_score = (70/100)*30 = 21.0
    raw_exam_score     = 58
    scaled_exam_score  = (58/100)*70 = 40.6
    grand_total        = 21 + 40.6 = 61.6 → ROUND to 62 → Grade: C4

  SCHOOL C (GES standard: class=50/50, exam=100/50):
    Components: Ex1=10, Ex2=8, Test=18, Project=14 (max: 10+10+20+10=50)
    raw_class_total    = 50
    scaled_class_score = (50/50)*50 = 50
    raw_exam_score     = 64
    scaled_exam_score  = (64/100)*50 = 32
    grand_total        = 82 → Grade: B2

8B.1 HOW PROMOTION ACTUALLY WORKS IN GHANAIAN SCHOOLS (RESEARCH BASIS)
--------------------------------------------------------------------------

Unlike the BECE/WASSCE grading scale (which IS nationally fixed by WAEC),
promotion from one class to the next WITHIN a school (e.g. Basic 4 -> Basic 5)
is NOT governed by one single rigid percentage handed down by GES. GES sets
the curriculum, the continuous-assessment (SBA) framework, and the academic
calendar (3 terms/year for basic schools), but day-to-day promotion decisions
are an internal school policy matter, built on top of that continuous
assessment data. This is why EduSphere implements promotion as a CONFIGURABLE
ENGINE (same philosophy as the Scoring Engine) rather than hard-coding one
formula — different schools genuinely use different thresholds and methods.

Patterns observed across Ghanaian basic schools and reflected in the engine:
  - Decision point: end of Term 3 (the final term of the academic year).
  - Basis: the pupil's performance across ALL THREE terms, not Term 3 alone.
  - Common method A — "Annual Average": average the Term 1, Term 2 and Term 3
    overall scores; pupil proceeds if the annual average clears the school's
    threshold (commonly 40-60%, school-defined).
  - Common method B — "Two of Three Terms": pupil proceeds if their average
    across any two of the three terms (one of which must be Term 3) clears
    the threshold — this gives some allowance for one bad term.
  - A pupil who is allowed to repeat a class is typically allowed to repeat
    that class ONCE; persistent failure triggers a Headteacher/PTA case
    conference rather than automatic repeated detention.
  - JHS 3 (Basic 9) pupils do not get "promoted" internally — they sit the
    BECE (set by WAEC) and are placed into SHS via CSSPS, based on their
    BECE aggregate (best six subjects, lower aggregate = better; pass mark
    is grade 6 = roughly 50% per subject; national qualifying aggregate
    guidance is commonly aggregate 36 or better, though placement also
    depends on CSSPS school choices and category).
  - SHS promotion from one form/year to the next is decided by the school
    based on continuous assessment + house/mock exams, plus conduct and
    attendance; WASSCE (also WAEC-set) at the end of SHS is a certification
    exam, not an internal promotion decision.
  - Tertiary institutions promote/progress students by credit accumulation
    and Cumulative Weighted Average (CWA)/GPA per semester rather than by
    "terms," so the Promotion Engine treats Tertiary as an out-of-scope
    variant exposed via the same configuration screen for completeness, but
    driven by credits-earned instead of term scores.

Because GES leaves the exact percentage and method to each school, EduSphere
ships with a sensible Ghana-aligned DEFAULT (Annual Average, threshold 40%,
repeat limit 1) but every figure is editable per school and per level.

8B.2 CORE CALCULATION LOGIC
------------------------------

Implemented as a Laravel Service class:
  App\Services\Promotion\PromotionEngineService

INPUT: For each student in a class, the already-published grand_total score
(from the Scoring Engine, Section 8) for Term 1, Term 2, and Term 3, across
all subjects (or a single configured "promotion subject set," e.g. core
subjects only, if the school chooses that variant).

CALCULATION FORMULA (Method 1 — Annual Average, the GES-aligned default):
  1. term1_overall = average of grand_total across all subjects in Term 1
  2. term2_overall = average of grand_total across all subjects in Term 2
  3. term3_overall = average of grand_total across all subjects in Term 3
  4. annual_average = ROUND(
        (term1_overall*w1 + term2_overall*w2 + term3_overall*w3)
        / (w1+w2+w3), decimal_places)
     -- w1, w2, w3 default to 1 (equal weight) but are school-configurable
  5. decision:
       IF annual_average >= promotion_threshold        -> "Promoted"
       ELSE IF annual_average >= conditional_threshold  -> "Conditional"
                                                            (probation, with
                                                            HOD/parent review)
       ELSE                                              -> "Repeat"

CALCULATION FORMULA (Method 2 — Two of Three Terms):
  1. Compute term1_overall, term2_overall, term3_overall as above.
  2. Select the pupil's best 2 of the 3 terms, REQUIRING Term 3 to be one of
     the two selected (configurable toggle: "Term 3 must be included").
  3. two_term_average = average of the 2 selected terms.
  4. Apply the same threshold comparison as Method 1.

CALCULATION FORMULA (Method 3 — Subject Pass Count):
  1. For the annual average subject scores, count how many subjects meet or
     exceed per_subject_pass_mark.
  2. decision:
       IF subjects_passed >= min_subjects_to_pass AND
          annual_average  >= promotion_threshold        -> "Promoted"
       ELSE                                              -> "Repeat" /
                                                            "Conditional"
       (Mirrors how some Ghanaian and West African schools combine a subject
       count rule with an overall average rule.)

EXAMPLE (Basic 4, Annual Average method, threshold 40%):
  Term 1 overall = 58.0
  Term 2 overall = 62.4
  Term 3 overall = 55.8
  annual_average = (58.0+62.4+55.8)/3 = 58.7
  58.7 >= 40  ->  Decision: PROMOTED to Basic 5

EXAMPLE (Basic 6, Two-of-Three method, threshold 50%):
  Term 1 overall = 38.0  (weak term)
  Term 2 overall = 54.0
  Term 3 overall = 57.0
  Best 2 of 3 INCLUDING Term 3: Term 2 (54.0) + Term 3 (57.0) = avg 55.5
  55.5 >= 50  ->  Decision: PROMOTED to Basic 7
  (Note: Term 1's poor score is forgiven by this method — exactly the
   allowance this method is designed to give.)

EXAMPLE (JHS 3 / Basic 9):
  exclude_terminal_year = true for this configuration
  -> Engine SKIPS internal promotion entirely.
  -> Student is tagged "BECE Candidate" and appears in the External
     Examinations module instead, pending WAEC's BECE results and CSSPS
     placement into SHS.

  - Configuration is versioned; each promotion_run stores a frozen
    rule_snapshot_json so that re-opening an old promotion list later still
    shows exactly which rule produced each decision, even if the school
    later changes its policy.

8B.4 PROMOTION RUN WORKFLOW (END OF TERM 3)
------------------------------------------------

  Route: /school/promotions/run?academic_year=2025-2026&level=primary

  1. GENERATE: Admin/Headteacher clicks "Generate Promotion List."
     - System requires all Term 3 scores for the level to be in "published"
       status (from the Scoring Engine workflow) before it will run.
     - Engine computes a decision for every student per Section 8B.2 and
       writes student_promotion_records (status: draft).

  2. PROMOTION LIST REVIEW SCREEN:
     | Student        | Class | T1   | T2   | T3   | Avg  | Decision   |
     |----------------|-------|------|------|------|------|------------|
     | Ama Serwaa     | 4B    | 58.0 | 62.4 | 55.8 | 58.7 | Promoted   |
     | Kofi Mensah    | 4B    | 32.0 | 38.5 | 35.0 | 35.2 | Conditional|
     | Yaw Boateng     | 4B    | 20.0 | 25.0 | 22.0 | 22.3 | Repeat     |
     | Adwoa Nyarko    | 9A    | —    | —    | —    | —    | BECE Cand. |
     Color flags: green=Promoted, amber=Conditional, red=Repeat,
     grey=Excluded (terminal year).
     Filter/sort by decision; export to PDF/Excel.

  3. CLASS TEACHER REVIEW:
     - Class teacher can add remarks per pupil ("recommend promotion despite
       borderline score due to improvement trend") and flag for override.
     - Status -> teacher_reviewed.

  4. OVERRIDE (optional, audited):
     - Any authorized role (Class Teacher proposes, Headteacher approves) can
       override a computed decision for an individual student.
     - Mandatory override_reason text field; both old and new decision are
       stored (is_override=true) for audit and cannot be silently changed
       later.

  5. HEADTEACHER/PRINCIPAL APPROVAL:
     - Reviews the full list and either [Approve] or [Send Back to Class
       Teacher] with comments.
     - Status -> approved.

  6. PUBLISH:
     - Status -> published.
     - System automatically:
         a. Creates next-year student_enrollments rows for all "Promoted"
            and "Conditional" pupils, with class_id = the configured next
            class and promoted_from_class_id = current class.
         b. Re-creates a same-class enrollment for "Repeat" pupils and
            increments student_repeat_history.
         c. Marks "BECE Candidate"/"WASSCE Candidate" students as pending
            in the External Examinations module rather than enrolling them
            automatically — their placement class is set once CSSPS/SHS
            admission or tertiary admission results are entered.
     - Promotion status/labels become visible on the Term 3 report card and
       on the parent portal ("Your child has been promoted to Basic 5" /
       "Your child is on academic probation in Basic 4" / etc.).

8B.5 SHS-SPECIFIC CONFIGURATION
------------------------------------

  SHS promotion configurations differ from basic school in these respects:
  - Subjects are split into Core (compulsory: English, Maths, Integrated
    Science/Social Studies, etc.) and Elective (programme-specific, e.g.
    General Science, Business, Visual Arts, Agriculture, Technical, Home
    Economics, General Arts).
  - Promotion rule can require: minimum average in CORE subjects AND minimum
    average in ELECTIVE subjects separately, not just one blended average.
  - Conduct and attendance gates are commonly added as additional
    promotion conditions (configurable boolean checks fed from the
    Discipline and Attendance modules).
  - SHS terminal year (final year before WASSCE) is automatically excluded
    from internal promotion, identical in spirit to JHS 3, and routed to the
    External Examinations (WASSCE) module instead.
  - House/mock exam results can optionally be blended into the SHS annual
    average via an extra configurable weight (e.g. Term scores 70% + Mock
    Exam 30%).

8B.6 INTEGRATION POINTS
----------------------------

  - Scoring Engine (Section 8): supplies the published per-term grand totals
    that the Promotion Engine consumes; promotion cannot run until scores for
    the level are published, preventing promotion decisions on incomplete
    data.
  - Report Card module: prints the promotion decision and the next class
    name on the Term 3 report card.
  - Parent Portal/SMS: notifies guardians of the promotion outcome.
  - External Examinations module: receives JHS 3/SHS-terminal pupils flagged
    "BECE Candidate"/"WASSCE Candidate" and later records WAEC results and
    CSSPS/tertiary placement, which then determines their actual next class
    or institution.
  - HR/Class Management: next-year class lists, streaming, and class-teacher
    assignment are auto-populated from the published promotion run.
  - Audit Log: every computed decision, every override, and every approval
    step is logged with actor, timestamp, and reason.

    gzip on;
    gzip_types text/css application/javascript application/json;
}

SECTION 17: BACKUP & RESTORE GUIDE
================================================================================

17.1 AUTOMATED BACKUP (Laravel Scheduler)
------------------------------------------

In App\Console\Kernel.php:
  $schedule->command('backup:run')->dailyAt('02:00');
  $schedule->command('backup:clean')->weekly();

Package: spatie/laravel-backup

Config: backup.php
  - Include: database + storage/app/public
  - Destination: local (for dev), S3/Wasabi (for production)
  - Password-protect zip: yes
  - Max age: 30 days (daily), 365 days (monthly)
  - Notification: email on failure

17.2 MANUAL BACKUP
-------------------

  # Database backup:
  mysqldump -u root -p edusphere_db > backup_$(date +%Y%m%d).sql

  # Files backup:
  tar -czf files_backup_$(date +%Y%m%d).tar.gz /var/www/edusphere/storage/app/public

17.3 RESTORE PROCEDURE
-----------------------

  Step 1: Stop queue workers:
    sudo supervisorctl stop edusphere-worker:*

  Step 2: Restore database:
    mysql -u root -p edusphere_db < backup_YYYYMMDD.sql

  Step 3: Restore files:
    tar -xzf files_backup_YYYYMMDD.tar.gz -C /

  Step 4: Clear caches:
    php artisan cache:clear
    php artisan config:cache

  Step 5: Restart workers:
    sudo supervisorctl start edusphere-worker:*

  Step 6: Test restore:
    - Log in as Super Admin
    - Verify school count matches pre-backup
    - Check a sample report card PDF renders
    - Verify website builder loads for a school

19.4 QUICK REFERENCE CARDS
---------------------------

CARD 1: Score Calculation Formulas
  Scaled Class = (Raw Class Total ÷ Class Max) × Class Weight
  Scaled Exam  = (Raw Exam Score ÷ Exam Max) × Exam Weight
  Grand Total  = Scaled Class + Scaled Exam

  Example (50/50 school):
    Class: 40/50 → (40÷50)×50 = 40
    Exam:  72/100 → (72÷100)×50 = 36
    Total: 40+36 = 76 → B2

CARD 2: GES Grading Scale (Standard)
  A1: 80-100 (Excellent)
  B2: 70-79  (Very Good)
  B3: 60-69  (Good)
  C4: 55-59  (Credit)
  C5: 50-54  (Credit)
  C6: 45-49  (Credit)
  D7: 40-44  (Pass)
  E8: 35-39  (Pass)
  F9: 0-34   (Fail)

CARD 3: Website Builder Keyboard Shortcuts

EXAMPLE 1: BASIC SCHOOL — ANNUAL AVERAGE METHOD (DEFAULT)
  Level: Primary, Class: Basic 4, Method: Annual Average

    Term 1 overall: 58.0   Term 2 overall: 62.4   Term 3 overall: 55.8
    Annual Average = (58.0+62.4+55.8)/3 = 58.7
    58.7 >= 40  ->  PROMOTED to Basic 5

EXAMPLE 2: BASIC SCHOOL — TWO-OF-THREE-TERMS METHOD
  Level: Primary, Class: Basic 6, Method: Two of Three (Term 3 required)
  Promotion Threshold: 50%
  Student: Kwabena Owusu
    Term 1 overall: 38.0   Term 2 overall: 54.0   Term 3 overall: 57.0
    Best 2 of 3 incl. Term 3: (54.0+57.0)/2 = 55.5
    55.5 >= 50  ->  PROMOTED to Basic 7

EXAMPLE 3: CONDITIONAL / BORDERLINE CASE
  Level: Primary, Class: Basic 4, Method: Annual Average
  Promotion Threshold: 40%, Conditional Threshold: 35%
  Student: Kofi Mensah
    Term 1 overall: 32.0   Term 2 overall: 38.5   Term 3 overall: 35.0
    Annual Average = 35.2
    35.0 <= 35.2 < 40  ->  CONDITIONAL (sent for HOD/parent review;
    Headteacher may approve promotion with a remediation plan, or convert
    to Repeat)

  Level: Primary, Class: Basic 3, repeat_limit = 1

    System flags: "Repeat limit reached — escalate to Headteacher case

    Status: "BECE Candidate" — routed to External Examinations module;

    BECE result and the CSSPS placement outcome, not by this engine.
